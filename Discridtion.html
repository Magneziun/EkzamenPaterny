4. Порождающие паттерны

4.1 Singleton · Одиночка

Где: Logger, ConfigManager
Зачем: Единственный экземпляр глобальных сервисов (логирование, конфигурация). Избегает конфликтов при обращении из разных частей системы.

4.2 Factory Method · Фабричный метод

Где: DeviceFactory, DroneFactory, CameraFactory
Зачем: Создание устройств (дрон, камера) без привязки к конкретным классам. Легко добавлять новые типы устройств.

4.3 Abstract Factory · Абстрактная фабрика

Где: ICityInfrastructureFactory, CityAFactory
Зачем: Создание семейств связанных объектов (транспорт + контроллер + сенсор) для разных регионов. Гарантирует совместимость.

4.4 Builder · Строитель

Где: BuildingBuilder
Зачем: Пошаговое конструирование сложного объекта (здание с множеством параметров). Отделяет сборку от представления.

4.5 Prototype · Прототип

Где: DevicePrototype, ConfigurableDevice
Зачем: Клонирование конфигураций устройств для быстрого создания похожих объектов без повторной инициализации.

5. Структурные паттерны

5.1 Adapter · Адаптер

Где: CameraAdapter
Зачем: Интеграция старого API камеры (OldCamera) в новый интерфейс INewCamera. Позволяет использовать устаревшее оборудование.

5.2 Bridge · Мост

Где: DeviceBridge, SmartLight, LocalControl, RemoteControl
Зачем: Разделяет абстракцию устройства и способ управления. Независимое развитие типов устройств и методов управления.

5.3 Composite · Компоновщик

Где: ICityComponent, City, District, SimpleBuilding
Зачем: Иерархия «город → район → здание → устройство». Единообразная работа с отдельными объектами и группами.

5.4 Decorator · Декоратор

Где: ServiceDecorator, LoggingDecorator
Зачем: Динамическое добавление поведения (логирование, кэширование, безопасность) без изменения кода. OCP.

5.5 Facade · Фасад

Где: CityFacade
Зачем: Простой унифицированный интерфейс для сложной подсистемы управления городом. Скрывает детали.

5.6 Flyweight · Приспособленец

Где: SensorType, SensorTypeFactory
Зачем: Переиспользование общих данных (типы сенсоров) для экономии памяти при тысячах устройств.

5.7 Proxy · Заместитель

Где: DeviceProxy
Зачем: Контроль доступа к устройству в зависимости от роли пользователя, ленивая загрузка, удалённые вызовы.

6. Поведенческие паттерны

6.1 Chain of Responsibility

Где: EventHandler, ValidationHandler, SecurityHandler
Зачем: Последовательная обработка событий (валидация → безопасность → логика). Гибкость порядка.

6.2 Command · Команда

Где: ICommand, TurnOnLightCommand, SendDroneCommand
Зачем: Инкапсуляция запросов в объекты. Очередь, логирование, отмена (undo).

6.3 Interpreter · Интерпретатор

Где: IExpression, GreaterThanExpression, Context
Зачем: Интерпретация DSL для правил («температура > 30 → охлаждение»). Пользовательские сценарии.

6.4 Iterator · Итератор

Где: DeviceCollection
Зачем: Единообразный обход коллекции устройств без раскрытия внутренней структуры.

6.5 Mediator · Посредник

Где: IMediator, EventMediator
Зачем: Централизованная координация между модулями. Снижает связанность.

6.6 Memento · Хранитель

Где: DeviceMemento, DeviceOriginator
Зачем: Сохранение и восстановление состояния устройства (откат перед опасной операцией).

6.7 Observer · Наблюдатель

Где: IObserver, SecurityService, EventPublisher
Зачем: Реакция на события (датчик движения → служба безопасности). Слабая связанность.

6.8 State · Состояние

Где: DeviceState, OnState, OffState, DeviceContext
Зачем: Поведение в зависимости от состояния (вкл/выкл/ошибка). Уход от switch.

6.9 Strategy · Стратегия

Где: IRoutingStrategy, FastestRoute, ShortestRoute, Navigator
Зачем: Выбор алгоритма маршрутизации (быстрый/короткий). Легкая замена.

6.10 Template Method

Где: EventProcessor, FireEventProcessor
Зачем: Скелет алгоритма обработки события (валидация → обработка → логирование) с переопределением шагов.

6.11 Visitor · Посетитель

Где: IVisitor, AuditVisitor, IVisitable, VisitableDrone
Зачем: Операции над объектами (аудит, аналитика) без изменения классов. Добавление новых операций без модификации иерархии.

6.12 Null Object · Пустой объект

Где: IHandler, NullHandler
Зачем: Замена null на объект с нейтральным поведением. Избавление от проверок на null.

7. Пример сценария работы: событие «Пожар»

📡 Как система реагирует на пожар (интеграция паттернов)

  • Генерация события: Датчик дыма (или внешняя система) создаёт событие "fire".
  • Chain of Responsibility: Событие проходит через ValidationHandler → SecurityHandler. Обработчик безопасности запускает команды.
  • Command: Выполняются SendDroneCommand (дрон для разведки) и TurnOnLightCommand (аварийное освещение).
  • Observer: EventPublisher уведомляет подписчиков (SecurityService → оповещение жителей).
  • Template Method: FireEventProcessor переопределяет шаг Handle(), добавляя вызов пожарных и эвакуацию, оставаясь в едином шаблоне.
  • Strategy: Navigator со стратегией FastestRoute прокладывает путь для пожарной машины.
  • State: Устройства (дрон, свет) переходят в OnState через DeviceContext.
  • Memento: Перед критической операцией состояние сохраняется через DeviceOriginator для возможного отката.
  • Visitor: После ликвидации пожара AuditVisitor собирает данные со всех задействованных устройств для отчёта.

✅ Такой подход обеспечивает слабую связанность, гибкость и расширяемость без изменения ядра системы.

8. Расширяемость архитектуры

  • ➕ Новый тип устройства → создать класс, реализующий IDevice, фабрику DeviceFactory, при необходимости адаптер.
  • 🧠 Новый алгоритм обработки → новая стратегия IRoutingStrategy или новый обработчик в цепочку обязанностей.
  • 🎮 Новый способ управления → новая реализация IControlMode через Bridge.
  • 📦 Новый сервис → реализовать IService и обернуть декораторами (логирование, кэш).
  • 📊 Новая операция над объектами → добавить нового посетителя (IVisitor) без изменения иерархии.

Все интерфейсы и абстракции размещены в соответствующих пространствах имён, упрощая поиск и интеграцию.

9. Обоснование выбора паттернов и избегание избыточной сложности

При проектировании соблюдены принципы SOLID и здравый смысл. Паттерны применены только там, где решают конкретную проблему:

  • Singleton — для глобальных сервисов (логгер, конфиг), где единственность критична.
  • Factory Method / Abstract Factory — создание устройств и семейств для разных регионов (требование ТЗ).
  • Builder — для зданий с множеством опциональных параметров, избегаем перегруженных конструкторов.
  • Prototype — клонирование конфигураций, ускоряет развёртывание устройств.
  • Adapter — интеграция старого оборудования без переписывания кода.
  • Bridge — разделение управления и устройства, предотвращает взрыв классов.
  • Composite — иерархия городских объектов, единообразная работа.
  • Decorator — сквозная функциональность (логирование) без засорения основных классов.
  • Facade — упрощает клиентский код.
  • Flyweight — экономия памяти для тысяч сенсоров.
  • Proxy — разграничение доступа (требование ТЗ).
  • Chain of Responsibility — гибкая обработка событий с возможностью изменения порядка.
  • Command — отложенное выполнение и откат (требование ТЗ).
  • Interpreter — поддержка пользовательских сценариев.
  • Iterator / Mediator / Memento — обход коллекций, снижение связанности, сохранение состояний.
  • Observer — реактивная архитектура, слабая связанность.
  • State — управление состояниями устройств без switch.
  • Strategy — смена алгоритмов маршрутизации.
  • Template Method — унификация pipeline обработки.
  • Visitor — добавление операций (аудит) без изменения иерархии.
  • Null Object — безопасная работа с опциональными компонентами.

Там, где достаточно простого конструктора или DI-контейнер избыточен, паттерны не навязывались искусственно. Все решения продиктованы конкретными требованиями к гибкости, сопровождаемости и производительности.
